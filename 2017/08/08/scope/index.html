遇到问题就要解决问题，如何解决问题？我们要透过现象看本质，要明白JS解析器的解析顺序，只有明白机理，才能更好地解决问题。
也许大家听说过“函数声明提前”，“变量声明提前”这些名词，其实这都JS解析机制的名词化，方便记忆而已。

JS解析器

JS代码执行阶段就是通过JS解析器完成的，该过程大致分为两个阶段：
1.预解析阶段（变量声明、函数声明）
在这个阶段，JS解析器在代码中寻找 var function关键字（函数中的参数也是一个var声明），并将这些声明提前，此时var声明的变量赋一个undefined，而函数声明，则是整个函数整体，这些声明都会放在最头部。注意：当遇到重名的变量和函数，函数会覆盖变量（因为函数有内容=_=,胖子欺负瘦子 ）
2.执行阶段（赋值运算、函数表达）
当预解析完成后，就要开始干活了，从上而下，执行表达式。

明白了原理，好，我们去分析上面那个题！
首先进行预解析，我们看到首先声明了一个var a，声明提前给它一个undefined，再声明一个函数a,该函数function a(){alert(2);}作为一个整体，声明提前并覆盖掉undefined，再往下走又声明一个函数a，该函数function a(){alert(4);}会把之前的内容覆盖掉。预解析到代码尾部了，OK，该阶段完成。进行执行阶段，首先alert(a)毋庸置疑，直接输出function a(){alert(4);}，然后给a赋值1，覆盖掉之前的函数，然后我们又遇到function a(){alert(2);}大家想一想a会不会变呢？肯定不会，因为这只是一个函数声明，声明已经在第一个阶段完成了。JS解析器直接跳过（除非是一个函数表达）之后 alert(a);输出1，之后又给a赋值3，输出3，之后又碰到一个函数声明，不care它，再次 alert(a);输出3。

怎么样？已经彻底明白了JS解析的机制了吧，这就是词法解析。 还没完，再做1个题目试试看？

输出 undefined 1 又蒙了？这个时候就牵扯到作用域的问题了，下文详细讲述一下作用域及其相关名词解释

JS有两个作用域。其一，全局作用域；其二，函数作用域。并不像C、JAVA中拥有块级作用域，但是ES6已经通过用let声明变量令循环表达式（例如for）
有了块级作用域的概念。当然这并不是本文的重点。还是回到JS全局与函数作用域这两个概念。

全局作用域：在全局声明了变量，这个变量在全局（Window环境）中都可用。

函数作用域：该函数在执行时，会拥有自己的作用域，在自己的作用域中仍然会执行预解析与执行阶段。函数中声明的变量，在本函数（包括内嵌
函数）可用，对外不可用（函数外部无法访问函数内部声明的变量）。

作用域链：当函数作用域内部对一个未声明的变量去赋值（例如a=2），在执行a=2表达式的时候，会在函数作用域中寻找声明的变量a，若找不到，则会
在上一级作用域中寻找是否有声明的a，若一直找不到，最终会在window声明该变量。这就组成了一个链式结构，这就是作用域链。

执行环境：即上下文环境，在表达式执行的时候，这个执行环境是动态的。本作用域内有声明变量，就会沿着作用域链找，所以说执行环境是动态的。

分析上面题目：
首先声明了a,又声明了一个函数, a=undefined , function fn(){alert(a);var a=2; } 都放在全局作用域头部。然后执行函数表达式，好的，fn函数
有自己的作用域，先预解析发现有一个a,将它提前，在然后执行阶段alert(a);输出undefined,再给a赋值2，函数执行完，a占用的内存被垃圾回收。然后
执行alert(a);因为函数作用域声明了自己的a,函数的执行环境只是在自己的作用域内，因此不影响全局a,因此输出1.
