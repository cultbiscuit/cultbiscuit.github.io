正文

遇到问题就要解决问题，如何解决问题？我们要透过现象看本质，要明白JS解析器的解析顺序，只有明白机理，才能更好地解决问题。也许大家听说过“函数声明提前”，“变量声明提前”这些名词，其实这都JS解析机制的名词化，方便记忆而已。

JS解析器

JS代码执行阶段就是通过JS解析器完成的，该过程大致分为两个阶段：
1.预解析阶段（变量声明、函数声明）
在这个阶段，JS解析器在代码中寻找 var function关键字（函数中的参数也是一个var声明），并将这些声明提前，此时var声明的变量赋一个undefined，而函数声明，则是整个函数整体，这些声明都会放在最头部。注意：当遇到重名的变量和函数，函数会覆盖变量（因为函数有内容=_=,胖子欺负瘦子 ）
2.执行阶段（赋值运算、函数表达）
当预解析完成后，就要开始干活了，从上而下，执行表达式。

明白了原理，好，我们去分析上面那个题！
首先进行预解析，我们看到首先声明了一个var a，声明提前给它一个undefined，再声明一个函数a,该函数functiona(){alert(2);}作为一个整体，声明提前并覆盖掉undefined，再往下走又声明一个函数a，该函数function a(){alert(4);}会把之前的内容覆盖掉。预解析到代码尾部了，OK，该阶段完成。进行执行阶段，首先alert(a)毋庸置疑，直接输出function a(){alert(4);}，然后给a赋值1，覆盖掉之前的函数，然后我们又遇到function a(){alert(2);}大家想一想a会不会变呢？肯定不会，因为这只是一个函数声明，声明已经在第一个阶段完成了。JS解析器直接跳过（除非是一个函数表达）之后 alert(a);输出1，之后又给a赋值3，输出3，之后又碰到一个函数声明，不care它，再次 alert(a);输出3。

怎么样？已经彻底明白了JS解析的机制了吧，这就是词法解析。 还没完，再做1个题目试试看？

输出 undefined 1 又蒙了？这个时候就牵扯到作用域的问题了，下文详细讲述一下作用域及其相关名词解释

JS有两个作用域。其一，全局作用域；其二，函数作用域。并不像C、JAVA中拥有块级作用域，但是ES6已经通过用let声明变量令循环表达式（例如for）有了块级作用域的概念。当然这并不是本文的重点。还是回到JS全局与函数作用域这两个概念。

全局作用域：在全局声明了变量，这个变量在全局（Window环境）中都可用。

函数作用域：该函数在执行时，会拥有自己的作用域，在自己的作用域中仍然会执行预解析与执行阶段。函数中声明的变量，在本函数（包括内嵌函数）可用，对外不可用（函数外部无法访问函数内部声明的变量）。

作用域链：当函数作用域内部对一个未声明的变量去赋值（例如a=2），在执行a=2表达式的时候，会在函数作用域中寻找声明的变量a，若找不到，则会在上一级作用域中寻找是否有声明的a，若一直找不到，最终会在window声明该变量。这就组成了一个链式结构，这就是作用域链。

执行环境：即上下文环境，在表达式执行的时候，这个执行环境是动态的。本作用域内有声明变量，就会沿着作用域链找，所以说执行环境是动态的。

分析上面题目：
首先声明了a,又声明了一个函数, a=undefined , function fn(){alert(a);var a=2; } 都放在全局作用域头部。然后执行函数表达式，好的，fn函数有自己的作用域，先预解析发现有一个a,将它提前，在然后执行阶段alert(a);输出undefined,再给a赋值2，函数执行完，a占用的内存被垃圾回收。然后执行alert(a);因为函数作用域声明了自己的a,函数的执行环境只是在自己的作用域内，因此不影响全局a,因此输出1.

明白了原理还想不想再做一些题目？好，下面分享三个题目，简单陈述重点，并给出结果，供大家参考。

输出： 1 2 原因：函数内部a沿着作用域链改变了全局变量a。

输出： undefined 1 原因：函数参数内的a 相当于声明了一个局部变量。

输出： 1 1 原因：函数参数内的a声明了， 并且fn(a)为函数传参了

好记性不如做笔记之JavaScript继承

发表于 2017-08-05

引言

我的处女博就献给JS继承吧哈哈哈^_^
正文开始之前先叨唠几句。前端学习可谓路漫漫其修远兮，但我们应有上下求索的精神。大体上说学习分四步，一是知识获取，二是知识巩固，三是知识运用，四是知识创新。而做笔记就是第二步的重中之重，这是一个理顺思路、整理知识的过程，将自己学到的知识点完整地写下来，进而加深印象，更好地学习。在健壮技术栈的路上做笔记可谓是功不可没。好了，下面总结一下近期对JS继承的理解。

—————并不华丽的废话分割线—————

在面向对象编程（OOP）中，继承是一个重要的概念。在讲继承之前，有必要去明白什么是构造函数、原型对象、原型链、prototype（proto）以及constructor属性。并且要透彻他们之间是什么关系。我一一对他们做简要介绍后期会博客专门阐述他们之间的关系。

构造函数：顾名思义构造函数是一个函数，但函数也是一个对象。它是创建对象的一种方法（通过new生成实例对象）例如内置对象Array就是一个构造函数，可以通过new Array()生成数组对象。

原型对象：每个构造函数初始化时都有一个prototype属性，构造函数.prototype就是原型对象

原型链 ：原型对象与构造函数通过prototype连接，而原型对象与实例对象（new 构造函数）通过proto连接，构造函数又通过proto与Function.prototype连接，而Function.prototype又通过proto与Object.prototype连接，还要考虑到在JS继承的子类又通过prototype属性与父类构造函数连接，我们可以看到这些实例对象、构造函数、原型对象、子类、子类的实例对象通过prototype与proto形成了一个链式结构，这个链就叫做原型链。（只是简单说一下，学问大着呢）

prototype：又称显式原型，存在于各个函数中。每个函数初始化时都有一个prototype属性，这个属性指向一个对象，即A.prototype（设函数为A）

proto : 又称隐式原型，存在于任何对象之中，每个人对象都有proto属性。该对象的proto指向它的构造函数的原型对象。

constructor：这个属性是原型对象自带的，这个属性是易变的，通常指向原型对象的构造函数。

—————这才是正文———————-

通过对以上名词的介绍，令大家对面向对象概念有个基本的了解。接下来就开始介绍JavaScript继承，目前掌握了三种继承方式，本文将通过实例详细总结继承的原理以及相应的优缺点。

第一种：原型链继承

该种方法的思想是：父类（原型对象）的实例作为子类的原型对象。话不多说直接扔代码：

function Dad(name){ this.name=name;}  //创建构造函数
Dad.prototype.getName=function(){console.log(this.name);}   //原型对象添加方法method

function Child(sex){this.sex=sex;}  //创建子类的构造函数
Child.prototype=new Dad("Scott");  //将Dad实例对象赋给子类原型（这是一个引用类型赋给另一个引用类型的过程）
Child.prototype.constructor=Child;//上面语句把new Dad的constructor赋给了Child.prototype
                                 //（当然该表达式不写也没有关系因为constructor是易变的，
                                //函数执行后又重新指向了Child。这么写是一种规范，应提倡！！
Child.prototype.ownMethod=function(){ console.log("my dad is"+this.name);}//子类增加方法

var foo=new Child("boy"); //实例化子集
foo.name;       //Scott  子类实例可以享用父类构造函数的属性name
foo.getName(); // >>Scott 子类实例可以享用父类的方法getName()
foo.sex;      // >>boy 子类自己的属性
foo.ownMethod();//>>my dad is Scott 子类实例调用自己的方法

var f=new Dad("Scott111");//创建一个父类的实例
f.ownMethod();  //>>error 可以看到父类并不能调用子类创建的方法！
f.getName();    //>>Scott111

这种父类引用类型赋给子类引用类型（父类实例对象赋给子类原型对象）的方法，使Child成功继承了Dad。实现父类方法的复用（子类没有该方法那么，子类对象就会沿着原型链向上找，直到找到这个方法，就用）

存在缺点：

1.当父类中包含引用类型属性的时候，这些引用类型会被实例对象共享，因为只有一个内存地址，实例对象将它改变了，在实例化其他的实例对象时，或者说其他子类在继承时，会有意想不到的后果。函数方法也是引用类型，也被共享，但不用担心被篡改，因为垃圾回收机制一个函数执行完，就会回收内部变量所占用的空间，而且函数作用域阻止了内部变量的共享。
2.上面代码Child.prototype=new Dad(“Scott”); 可以看到，在创建子类原型时，父类的实例对象已经将参数传递了，以后子集实例化对象时并不能給父类构造函数传参。

第二种：构造函数继承（call apply借用）

function Dad(name){ this.name=name;}  //创建构造函数
Dad.prototype.getName=function(){console.log(this.name);}   //原型对象添加方法method

function Child(sex,name){this.sex=sex;Dad.call(this,name);}   //创建子类的构造函数,利用call将父类构造函数的中this指向了子类构造函数（借用）
Child.prototype.constructor=Child; 
Child.prototype.ownMethod=function(){ console.log("my dad is"+this.name);}//子类增加方法

var foo=new Child("boy","Scott's son"); //实例化子集
foo.name;       // >>Scott's son 子类实例子集传参用自己的name
foo.getName(); //>>error 无法共享父类的方法！！！！！！
foo.sex;      // >>boy 子类自己的属性
foo.ownMethod();//>>my dad is Scott's son  =。=

var f=new Dad("Scott111");//创建一个父类的实例，传入自己的参数，不受子类传参影响
f.ownMethod();  //>>error 可以看到父类并不能调用子类创建的方法！           
f.getName();   //>>Scott111 输出父类名字

这种继承方法通过call（或者apply)将父类构造函数的this指向了子类构造函数，这样name就相当于Child的属性了。因为是继承了构造函数所以解决了第一种方法中的缺点。即将属性和方法都写在构造函数里，子类生成一个对象就会调用父类构造函数生成新的方法和属性，因此方法没有实现共享。因为没有了引用类型被改变的缺点，同时也可以向构造函数传参。

存在缺点：

1.父类所有方法和属性都要写在构造函数里，每个实例对象都会拥有一个方法，没有实现方法服用，降低性能。
2.父类定义的原型方法对子类不可见（因为子类原型是被构造函数赋值的，父类Dad.prototype被隔离了）

第三种：原型链+构造函数组合继承

function Dad(name){ this.name=name;}  
Dad.prototype.getName=function(){console.log(this.name);}   

function Child(sex,name){this.sex=sex;Dad.call(this,name);}  
Child.prototype=new Dad();    
Child.prototype.constructor=Child; 
Child.prototype.ownMethod=function(){ console.log("my dad is "+this.name);}

var foo=new Child("boy","Scott's son"); 
foo.name;       // >>Scott's son 
foo.getName(); // >>Scott's son 
foo.sex;      // >>boy 
foo.ownMethod();//>>my dad is Scott's son //不和谐的输出=。=

var f=new Dad("Scott111");
f.ownMethod();  //>>error            
f.getName();   //>>Scott111

相信大家看懂了上面两种继承，这种组合式的继承模式理解起来就小case了。这种方法克服了方法一、二的缺点
存在缺点：
内存占用大了，每次调用子类实例时，父类构造函数会执行两次，浪费性能

补充：
除此之外，继承方法还有原型式继承、寄生式继承以及寄生组合式继承等，就不一一介绍了。
